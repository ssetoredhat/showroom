== Kafka基礎 - Topic ReplicationとFault Tolerance
Kafkaのフォールトトレランス（耐障害性）は、基本的にtopic replicationメカニズムに基づいて構築されており、broker障害が発生した場合にデータの耐久性と高可用性を保証します。

各partitionについて、Kafkaは異なるKafka brokerに分散されたreplicaと呼ばれる複数のコピーを維持します。

** 1つのreplicaがleaderに指定されます。そのpartitionのすべてのproduceとconsumeリクエストは、強力な一貫性とメッセージの順序付けを保証するために、leaderによって排他的に処理されます。
** 他のreplicaはfollowerです。彼らの唯一の責任は、最新の状態を保つためにleaderからメッセージを熱心にreplicationすることです。

** leader partitionを持つbrokerがダウンすると、新しいleader partitionが別のノードで選出されます。

実際に動作を見てみましょう。



* **上部のターミナル** で、replication factorが3の新しいtopicを作成します。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-topics.sh --bootstrap-server $BOOTSTRAP \
  --create --topic replicated --partitions 3 --replication-factor 3
----

* https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"] > topics > replicatedに移動し、topic `replicated`が3つのpartitionで作成されていることを確認します。
+
image::m1/replicated-topic.png[]

* topic名`replicated`をクリックし、partitionsタブに移動してtopicの詳細を表示します。各partitionに1つのleaderと2つのreplicaがあることがわかります。
+
image::m1/replicated-topic-partitions.png[]

* 括弧内に表示されている番号はbroker idです。この場合、id 0、1、2の3つのbrokerがあります。leaderとreplicaは、フォールトトレランス（耐障害性）を確保するためにこれらのbrokerに分散されています。

* 各partitionには1つのleaderと2つのfollower（replica）があります。leaderはそのpartitionのすべての読み取りおよび書き込みリクエストを処理し、followerはフォールトトレランス（耐障害性）を確保するためにleaderからデータをreplicationします。

* leader brokerがダウンすると、followerの1つが自動的に新しいleaderとして選出され、高可用性が確保されます。これにより、Producerとconsumerは中断することなく動作を継続できます。

NOTE: この演習ではbroker障害をシミュレートしません。モジュールの残りの部分で稼働している必要があるためです。


=== まとめ
* フォールトトレランス（耐障害性）は、topic replicationメカニズムに基づいて構築されています。これにより、broker障害が発生した場合にデータの耐久性と高可用性が保証されます。

* Replication factorは、各partitionにフォールトトレランス（耐障害性）を確保するための1つのleaderと複数のfollower（replica）があることを保証します。

* replicaは異なるbrokerに分散されてフォールトトレランス（耐障害性）を確保するため、1つのbrokerがダウンしても、データは他のbrokerで引き続き利用できます。
