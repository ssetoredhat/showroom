== アクティビティ


* _recommendation engine_アプリケーションがおすすめ製品のリストを計算するために使用しているコードを見てみましょう。コードを見るにはここをクリックしてください: https://github.com/rh-cloud-architecture-workshop/recommendation-engine/blob/main/src/main/java/org/globex/retail/streams/TopologyProducer.java[TopologyProducer.java, window=”_blank”]

* *Kafka topic trackingEventTopicからの入力ストリームを下流処理用に準備する (Line 45 - 46)* 。これがパイプラインのエントリーポイントです
+
[source,java]
----
KTable<String, ProductScore> productLikes =
    builder.stream(trackingEventTopic, Consumed.with(Serdes.String(), Serdes.String()));
----

** Kafka TopicからStreamを作成 - `builder.stream(...)` は、Kafka Streamsに対してKafka topicから連続的なレコードフローの読み取りを開始するよう指示します。

** Serializers/Deserializers - `Consumed.with(Serdes.String(), Serdes.String())` は、Kafka Streamsにデータの読み取り方法を指示します。ここでは、keyとvalueの両方がプレーンな文字列です。


* *変換と集約(Line 47-51)* - このブロックは、生のユーザーアクティビティメッセージを取得 → 解析 → 無関係なものをフィルタリング → 製品スコアに変換 → 製品ごとにグループ化 → 最後に「いいね」を集約します。
+
[source,java]
----
.mapValues(JsonObject::new)
.filter((key, value) -> value.getJsonObject(“actionInfo”) != null
    && value.getJsonObject(“actionInfo”).getString(“productId”) != null)
.mapValues(value -> new ProductScore.Builder(
    value.getJsonObject(“actionInfo”).getString(“productId”)).build())
.groupBy((key, value) -> value.getProductId(),
    Grouped.with(Serdes.String(), productLikesSerde))
.reduce(ProductScore::sum)

----
** データの変換とクリーニング - 各JSON文字列をJsonObjectに変換し、有効な製品IDを持たないレコードを除外します。

** ProductScoreオブジェクトの作成 - 各メッセージに対して新しいProductScoreオブジェクトを作成します。

** 製品ごとにグループ化 - 同じ製品のすべてのイベントが一緒になるようにストリームを再グループ化します。舞台裏では、Kafka Streamsが関連するレコードをシャッフルおよびグループ化するための内部topicを作成します。

** Reduce（スコアの集約） - 各製品のスコアの累計をstate storeに保持します。結果は、すべての製品の現在の人気度を示すKTableです。

* *すべての製品スコアの更新をリスニングし、結果を自動的に保存・更新することで、カテゴリごとのTop-Nおすすめ製品を継続的に追跡する (Line 53-57)*
+
[source,java]
----
productLikes
  .groupBy((key, value) -> KeyValue.pair(value.getCategory(), value),
           Grouped.with(Serdes.String(), productLikesSerde))
  .aggregate(
      () -> new FixedSizePriorityQueue<>(ProductScore.SCORE_ORDER, aggregationSize),
      (key, value, aggregate) -> aggregate.add(value),
      (key, value, aggregate) -> aggregate.remove(value),
      Materialized.<String, FixedSizePriorityQueue, KeyValueStore<Bytes, byte[]>>as(aggregationStore)
          .withKeySerde(Serdes.String())
          .withValueSerde(fixedSizePriorityQueueSerde)
  );

----

** カテゴリごとにグループ化 - 同じカテゴリのすべての製品が一緒にグループ化されるようにストリームを再キー化します。カテゴリごとの集約のためにデータを準備します。

** Aggregate: カテゴリごとのTop-Nリストを構築

*** Initializer: 上位スコアの製品のみを保持する空の固定サイズキューを作成します。

*** Adder: 新しいスコアが到着したときに、そのキュー内の製品を追加または更新します。

*** Subtractor: 製品のスコアが変更された場合、古いエントリを削除します（結果を正確に保ちます）。

*** 各カテゴリには、リアルタイムで更新される独自の「Top N製品」リストがあります。

** Storeへのマテリアライズ - これらのカテゴリごとのトップリストをaggregationStoreという名前のローカルstate storeに保存します。Kafka Streamsは、アプリの再起動時に状態を復元できるように、changelogもKafkaに書き込みます。

=== まとめ
まとめると、Kafka Streamsはアプリケーションが1つ以上の入力ストリームからデータを受信できるようにします。このAPIを使用して、マッピング、フィルタリング、結合などのリアルタイム操作のシーケンスをデータストリーム上で実行できます。これが、Kafka Streamsがデータの流れに応じて処理し反応する方法を理解するのに役立てば幸いです。Kafka Streamsは、リアルタイムメッセージングと継続的な計算の間のギャップを埋め、シンプルなKafkaイベントのストリームをライブで進化する洞察に変えます。
